Rhythm Versioning System (RhyVer)

Rhythm versioning system helps teams build a release cadence. Using the format vMAJOR.YYQ#.WEEK.PATCH, it combines semantic versioning principles with granular temporal context, creating a natural rhythm for product releases. For example, v2.24Q4.45.32 indicates a major version 2 release from the year 2024, quarter 4, week 45, with 32 backward compatible changes, enabling predictable release cycles and better alignment with business goals.

Given a version vMAJOR.YYQ#.WEEK.PATCH, increment the:

  • MAJOR version when you make API compatibility changes.
  • YYQ# to match the current year and quarter.
  • WEEK number to match the ISO week (1-53).
  • PATCH for backward compatible changes that do not change the API.

Rhythm Versioning Specification

vMAJOR.YYQ#.WEEK.PATCH[-PRERELEASE][+BUILD]
RhyVer Example

v2.24Q4.45.32-rc.2


  Example: v2.24Q4.45.32-rc.2
            │ │    │  │  │
            │ │    │  │  └──  Release candidate
            │ │    │  └─────  Patch
            │ │    └────────  ISO week
            │ └─────────────  Year & quarter
            └───────────────  Major version
Components
01. MAJOR
Non-negative integer with a prefix “v”.
Changes for incompatible API modifications.
Example: v2 indicates a major version 2.
02. YYQ#
YY: Two-digit year (23, 24, etc.).
Q#: Quarter number (Q1-Q4).
Provides fiscal/planning context.
Example: 24Q2 = 2024, Quarter 2
03. WEEK
ISO week number (1-53).
Maintains global standard.
Continuous through year.
Example: Week 18 (Standard ISO week)
04. PATCH
Non-negative integer.
Fine-tuning within the week.
Incremental updates and bug fixes.
Example: 1
05. PRERELEASE (Optional)
Optional preview releases.
Format: -alpha.1, -beta.1, -rc.1
Example: v2.2024.43.0-beta.1
06. BUILD (Optional)
Optional build metadata.
Format: +build.123
Example: v2.2024.43.0+build.123

Rhythm Version Validation

Here's a simple regex pattern for validating rhythm version strings:


        ^v(0|[1-9]\d*)\.(2\d)Q([1-4])\.((?:[1-9]|[1-4]\d|5[0-3]))\.(0|[1-9]\d*)(?:-((?:0|[1-9]\d*|\d*[a-zA-Z-][0-9a-zA-Z-]*)(?:\.(?:0|[1-9]\d*|\d*[a-zA-Z-][0-9a-zA-Z-]*))*))?(?:\+([0-9a-zA-Z-]+(?:\.[0-9a-zA-Z-]+)*))?$

Enforcing:

  • Prefix 'v' and MAJOR version. Examples: v0, v1, v42
  • YYQ# format. Examples: 24Q1, 25Q3
  • ISO week number. Examples: 1, 42, 53
  • PATCH version. Examples: 0, 1, 42
  • Pre-release Suffix (Optional). Examples: -alpha.1, -beta.2, -rc.1
  • Build Metadata (Optional). Examples: +build.123, +exp.sha.5114f85

FAQs

How do I handle ISO week vs Quarter misalignment?

Use the business quarter designation that makes most sense for your reporting, regardless of exact ISO week boundaries. For example:


        v2.24Q1.13.0  // Even if week 13 slightly overlaps Q2
        v2.24Q2.14.0  // Starting Q2 designation with week 14

The quarter designation (Q#) is primarily for business context, while the ISO week number provides the precise temporal reference. This intentional flexibility allows for business alignment while maintaining technical accuracy.

If I'm releasing a version that was developed across a quarter boundary, which quarter should I use?

Always use the quarter in which the release is happening, not when development started. For example:


        v2.24Q1.13.2  // Last release in Q1
        v2.24Q2.14.0  // First release in Q2, even if development started in Q1

This maintains clarity about when the software was actually released, which is more valuable than tracking development time.

How do I handle hotfixes vs regular patches?

Both hotfixes vs regular patches use the PATCH number, but they can be distinguished by the release process:


        Regular patch: v2.24Q2.16.0 → v2.24Q2.16.1  // Normal iteration
        Hotfix from production: v2.24Q2.16.0 → v2.24Q2.16.1-hotfix.1 → v2.24Q2.16.1  // Emergency fix

Use pre-release tags for hotfix candidates if they need testing before full deployment.
